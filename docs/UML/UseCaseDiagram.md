## 1 目标

设计UML用例图往往出于以下的目标：

- **描述系统功能**：清晰地展示系统能够提供的主要功能。
- **识别参与者**：明确系统与哪些外部实体进行交互。
- **定义交互关系**：展示参与者如何使用系统的功能。
- **辅助需求分析**：帮助开发团队和利益相关者理解和确认系统需求。



## 2 组成

用例图主要由以下几个元素组成：

- **参与者（Actor）**：
  - **定义**：参与者是与系统交互的外部实体，可以是人、其他系统或硬件设备。
  - **表示**：通常用**人形图标**表示。
- **用例（Use Case）**：
  - **定义**：用例表示系统能够执行的一个功能或一组相关功能。
  - **表示**：通常用**椭圆形**表示。
- **关联（Association）**：
  - **定义**：表示参与者与用例之间的交互关系。
  - **表示**：用**实线**连接参与者和用例。

!!! example "关联示例"

    ![关联示例](./assets/image-20251128143005798.png)

- **包含关系（Include）**：
  - **定义**：表示一个用例在执行过程中会包含另一个用例的功能。用于向基础用例中添加没有明确规定的功能。
  - **表示**：用**带箭头的虚线**连接，箭头指向被包含的用例，并标注**<< Include >>**。

!!! example "包含关系示例"

    ![包含关系示例](./assets/image-20251128144512378.png)

- **扩展关系（Extend）**：
  - **定义**：扩展关系表示一个用例（扩展用例）在特定条件下会扩展另一个用例（基本用例）的功能。
  - **表示**：用**带箭头的虚线**连接，箭头指向被扩展的用例，并标注**<< Extend >>**。

!!! example "扩展关系示例"

    想象一下，你有一个基本的操作，比如 **登录账户**。有时候，根据不同的情况，你可能需要做一些额外的操作。这些额外的操作不是每次都会发生，而是在特定条件下才会执行。这就是 **扩展关系** 的作用。
    
    **扩展关系** 就像是给一个基本操作添加了一些 **"如果……那么……"** 的分支。这些分支操作是 **可选的**，只有在满足特定条件时才会执行。
    
    假设我们有一个 **登录账户** 的功能：
    
    - **基本操作：登录账户**
      - 用户输入用户名和密码，尝试登录。
    
    现在，根据密码是否正确，我们有两种可能的情况：
    
    1. **密码有效**
       - **描述**：如果用户输入的密码是正确的，那么用户成功登录。
       - **条件**：密码验证通过。
    2. **密码无效**
       - **描述**：如果用户输入的密码是错误的，系统会提示用户密码错误，并可能要求用户重新输入。
       - **条件**：密码验证失败。
    
    ![扩展关系示例](./assets/image-20251128143250691.png)

- **泛化关系（Generalization）**：
  - **定义**：表示参与者或用例之间的继承关系。
  - **表示**：用**带空心三角箭头的实线**连接，箭头指向父类。
