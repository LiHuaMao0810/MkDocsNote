    测试用例

    测试生命周期

    测试用例的产生不是孤立的，它贯穿于从需求分析到报告生成的整个过程。核心步骤包括测试目标设定、估算、策略制定、进度安排，以及具体的需求可测试性分析、用例设计、执行、缺陷跟踪和最终的总结报告。

    Example

    在开发外卖 APP 的“搜索”功能时，测试生命周期从拿到需求文档那一刻就开始了。QA 需要先分析搜索框是否支持模糊查询（需求分析），然后决定是手动点点点还是写自动化脚本（策略），接着编写输入各种关键词的用例（设计），最后在手机上运行并记录发现的 Bug（执行与报告）。

    测试生命周期（Test Life Cycle）描述了测试活动从项目启动到最终交付的完整演进过程，它确保了测试工作是有计划、有组织且可追溯的。

    测试生命周期的核心阶段

    测试并不是在代码写完后才开始的动作，而是一系列并行于开发的任务流。它包含了从前期的测试目标设定、资源估算、策略制定，到中期的需求分析、环境搭建、用例设计，再到后期的测试执行、缺陷跟踪以及最终的覆盖率分析与报告提交 。

    Example

    假设你要测试外卖 APP 的“余额支付”功能。

    在分析阶段，QA 就要介入确认：如果余额刚好等于订单价格能不能付？如果支付瞬间网络断了钱扣不扣？（需求分析与可测性分析）。

    在设计阶段，你要写好“余额充足支付”和“余额不足支付”的具体操作步骤（用例设计）。

    在执行阶段，你打开手机实际操作并记录扣款是否正确（测试执行）。最后，如果发现余额被扣成负数了，你要提交 Bug 并跟着程序员修好它（缺陷跟踪与报告）。

    测试目标（Test Objective）与策略（Test Strategy）

    在生命周期的起始阶段，必须明确“为什么测”以及“怎么测”。这包括定义测试的深度、选择合适的测试工具，并预估所需的时间和人力 。

    Example

    针对 APP 的“双十一大促”活动，测试目标设定为“确保系统能承受每秒 1 万次的下单请求”。测试策略则决定：因为人工点手机测不了高并发，所以必须采用“自动化性能测试”策略，使用专门的压测工具来模拟流量，而不是靠人手。

    需求可追溯性（Requirement Traceability）

    在整个生命周期中，必须确保每一个原始需求都能在末端的测试用例中找到对应。这种金字塔逻辑保证了测试覆盖的完整性，防止功能遗漏 。

    缺陷跟踪（Defect Tracking）与分析

    当测试执行发现实际结果与预期不符时，必须进入缺陷生命周期管理。这不仅仅是记录 Bug，还包括分析 Bug 产生的趋势，以此评估软件是否达到了可以发布的质量水平 。

    Example

    在测试执行阶段，你发现“配送员位置不刷新”。你提交一个 Bug 报告，开发将其状态改为“已修复”，你再进行“回归测试”确认。如果生命周期末期发现 Bug 的曲线已经明显下滑并趋于平稳，说明系统变稳定了，可以准备出具测试报告并结束本次生命周期。

    测试用例设计目标

    好的测试用例需要满足三个核心目标：良好的测试覆盖率（确保该测的都测到了）、良好的可追溯性（每个用例都能对应到需求点）以及良好的可执行性与可维护性（步骤清晰且容易根据版本变化进行更新）。

    Example

    针对“用户登录”功能，如果你的用例只测了“输入正确密码登录”，那覆盖率就不够。你还需要设计“密码错误”、“手机号未注册”、“验证码过期”等用例。同时，这些用例要在文档中明确标注对应的是需求说明书中的“登录模块第 2.1 条”，方便后续需求变更时快速找到并修改。

    测试用例设计目标是衡量测试工作质量的关键准绳，它决定了测试用例是否能够高效、准确地发现问题并长期维护。

    良好的测试覆盖率（Good Test Coverage）

    测试用例必须能够完整地覆盖所有的业务需求、功能特性以及可能的逻辑分支，确保没有遗漏任何一个死角。

    Example

    在测试外卖 APP 的“下单”功能时，如果你的测试用例只包含了“在线支付成功”这一种情况，覆盖率显然是不够的。一个目标明确的测试设计还应覆盖：余额不足、支付超时、商家满单无法接单、优惠券过期等所有可能的业务逻辑，确保系统在各种极端场景下都能正常反馈。

    可执行性与可维护性（Executability and Maintainability）

    测试用例的组织结构应当合理，步骤描述应当清晰无歧义，以便不同的测试人员都能顺利执行。同时，当软件发生变更时，用例应当易于更新，以最小化维护成本。

    Example

    假设“登录”功能的步骤写成“尝试登录”，这就是一个不可执行的差用例。好的设计应该是：

    1. 输入手机号 138xxx；2. 点击获取验证码；3. 输入验证码 1234；4. 点击登录按钮。

    这样即使是新入职的同事也能照着做。此外，如果 APP 增加了“图形验证码”环节，你只需要修改这一处步骤，而不需要推翻重写整个登录用例。

    良好的可追溯性（Good Test Traceability）

    每一个测试用例都必须能追溯到具体的功能点或用户需求，确保“测有所依”。这种纵向的链接关系可以帮助团队清晰地看到哪些需求已经测过，哪些还没测。

    关注最终用户需求（Keep End User Needs in Mind）

    测试设计不应仅仅局限于技术文档的字面意思，而应站在最终用户的角度，思考用户在真实场景中会如何使用产品。

    Example

    需求文档说“图片上传不能超过 5MB”。从技术上看，你测 5.1MB 报错就够了。但从用户角度出发，你还需要设计：在信号极弱的电梯里上传图片、上传过程中突然来电话、上传一半手机断电等场景。因为这些才是用户在现实生活中最常遇到的“糟心事”，也是最容易暴露质量问题的时刻。

    测试用例设计方法

    等价类划分法（Equivalence Class Partitioning, ECP）

    这是一种典型的黑盒测试方法。其核心理论是将程序的输入域划分为若干部分，从每个部分中选取少数代表性数据进行测试。它基于“单故障假设”，即子集中的任何一个元素产生的测试结果都应该是相同的。划分时分为有效等价类（符合规格的输入）和无效等价类（不符合规格的非法输入）。

    Example

    假设一个文本框只接受 2 到 10 个大写字母。

    有效等价类：长度在 2-10 之间的 A-Z 字符串（如 "HELLO"）。

    无效等价类 1：字符不在 A-Z 范围内（如 "123"）。

    无效等价类 2：长度小于 2（如 "A"）。

    无效等价类 3：长度大于 10（如 "ABCDEFGHIJKL"）。

    边界值分析法（Boundary Value Analysis, BVA）

    历史经验表明，大量的错误往往发生在输入范围的边界及其附近。BVA 关注的是输入域的边缘，通常包括最小值、略大于最小值、标称值、略小于最大值、最大值。此外，还包括健壮性测试（即超出边界的正负一个单位）。

    Example

    同样是要求“长度 2 到 10 个字符”的文本框。

    边界值测试点包括：1（最小值-1）、2（最小值）、3（最小值+1）、9（最大值-1）、10（最大值）、11（最大值+1）。测试人员会专门输入只有 1 个字母或刚好 11 个字母的情况，看系统是否会报错。

    组合分析/两两组合法（Combinatorial Analysis / Pairwise）

    当多个参数相互影响时，所有组合的数量会呈指数级爆炸。Pairwise 理论认为，大多数 Bug 是由单参数或两个参数的交互触发的。因此，该方法通过设计一个极小的子集，确保任意两个参数的所有可能取值组合都至少被覆盖一次，从而在缩减测试成本的同时保持较高的信心。

    Example

    测试视频录制功能，有 4 种编码格式、4 种分辨率、4 种帧率。如果全测需要 \(4 \times 4 \times 4 = 64\) 组实验。使用两两组合法，工具会自动筛选出约 16 组用例，保证“H.264 格式配 1080P”、“MPEG-4 格式配 25fps”等任意两项组合都被测到，大大减少了工作量。

    错误推测法（Error Guessing）

    这是一种基于经验和直觉的非正式设计方法。测试人员根据以往的项目经验、对业务逻辑的深刻理解以及对开发人员容易犯错环节的判断，列举出程序中可能存在的缺陷或容易出问题的特殊场景。

    Example

    经验丰富的 QA 在测支付功能时，会故意在支付过程中点击“返回”键、频繁连续点击“支付”按钮、在手机电量低于 1% 关机瞬间进行支付。这些场景在需求文档里可能没写，但根据经验，这些“骚操作”最容易导致系统出现重复扣款或订单状态异常。
